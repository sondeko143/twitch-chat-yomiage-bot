# 秘密・設定を OS 標準ユーザーディレクトリへ

## 問題

`client_secret` を含む秘密・設定が、CWD の `.env` から平文で読み込まれる（`dotenvy::dotenv()` + `config` の `cb_` プレフィックス env 経由）。このため 2 つの困りごとがある:

- **CWD 依存**: 起動ディレクトリに `.env` が無いと動かず、起動場所に挙動が縛られる。トークン DB（`db/data.json`）も CWD 相対で同じ問題を持つ。
- **リポジトリ作業ツリーへの秘密の同居**: 平文の秘密値がリポジトリの作業ツリー内に置かれるため、`.gitignore` で守っていても誤コミット・履歴混入の余地が残る（構造的に不可能ではない）。

## ゴール

秘密・設定・トークンの保存先が、リポジトリ作業ツリーの外にある OS 標準のユーザーディレクトリに固定され、どの CWD から起動しても同一に動作する。

## 非ゴール

- at-rest 暗号化 / OS 資格情報ストア（keyring）への格納。今回は平文ファイルのまま、場所だけを移す。
- マルチプロファイル、シークレットの自動ローテーション。
- 既存のリーク検出ゲート（gitleaks / check-env-leak / `.gitignore`）の仕様変更。

## 受入基準

- [ ] 任意の CWD から起動しても、同一の設定とトークンが読まれる（CWD の `.env` の有無に挙動が依存しない）。
- [ ] 秘密値を含む設定ファイルとトークン DB のデフォルト保存先が、リポジトリ作業ツリーの外の OS 標準ユーザーディレクトリである。
- [ ] 設定ファイルが存在しない初回起動時、プレースホルダ入りテンプレが OS 標準パスに生成され、その絶対パスと「編集して再実行」の案内が表示されて終了する。
- [ ] shell 環境変数によるオーバーライドと `--config <path>` の明示指定が、引き続き設定値に反映される。
- [ ] 既存利用者向けの移行手順（既存 `.env` / `db/data.json` の扱い）が README 等に文書化されている。
- [ ] `just ci` が全緑（exit code 0）。
